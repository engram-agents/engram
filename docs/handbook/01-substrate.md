# ENGRAM Substrate Layer (L0) — Mechanisms & Architecture

Based on comprehensive archaeology of the `engram-alpha` repo, here is the mechanism inventory for the technical handbook:

## **1. ID Generation & Sequencing**

**what** — Atomic, prefixed ID generation for all node types. Each node type (observation, derivation, axiom, etc.) maintains a monotonic counter in the `sequences` table; new node creation atomically increments its counter and returns a zero-padded ID (`ob_NNNN` shape). IDs are immutable, human-readable, and grow unbounded past the 4-digit zero-padding boundary.

**how** — Entry point: `_next_id()` in `engram_core.py:2328` (server.py wrapper at :350). Key data structure: `sequences(node_type TEXT PRIMARY KEY, last_num INTEGER)` table. Atomic operation uses SQLite's `INSERT...ON CONFLICT...DO UPDATE` returning the incremented value. ID format: `f"{prefix}_{num:04d}"` where prefix is looked up from `TYPE_PREFIX` dict in `engram_core.py:519` (server.py re-exports as alias). Validation regex in `engram_ids.py` (NODE_ID_RE: `\b([a-z]{2}_\d{4,})\b`).

**status-candidate** — **prod-presumed** (core mechanism, exercise in every tool that creates nodes).

**files** — server.py (3071–3086), engram_ids.py (complete)

**key constants** — `TYPE_PREFIX` dict entries: `"observation_factual": "ob"`, `"derivation": "dv"`, `"axiom": "ax"`, `"definition": "df"`, `"goal": "gl"`, etc. (`engram_core.py:519`; server.py re-exports as alias).

**tests** — Implicit in all node-creation tests (test_bootstrap_fresh_db.py, test_engine_flows.py, etc.). Explicit validation in engram_ids.py doctest examples.

## **2. Schema & Migrations**

**what** — Foundational SQLite schema (nodes, edges, sequences) with append-only migration pathway for retroactive column additions. New columns (memory status, embeddings, focus state, feeling reports, question metadata, trust tiers) are added dynamically via `ALTER TABLE` on startup, with idempotent backfill operations for each new column (e.g., trust_tier backfilled to "unknown" for all person nodes).

**how** — Entry point: `_get_db()` in `engram_core.py:1100` (server.py wrapper at :269). Core tables created in `engram_core.py` (DB init section ~1100–1620):
- `nodes(id TEXT PRIMARY KEY, type, claim, created_at, ... [90+ columns])` — central knowledge store.
- `edges(source_id, target_id, relation, created_at, UNIQUE constraint)` — directed graph.
- `sequences(node_type PRIMARY KEY, last_num)` — ID counters.

FTS5 index: `nodes_fts(claim, quoted_text, interpretation, content='nodes')` (external-content virtual table, `engram_core.py` DB init). SQLite-vec KNN index: `vec_nodes(node_id PRIMARY KEY, embedding float[384])` (`engram_core.py` DB init) when _VEC_BACKEND_AVAILABLE.

Diagnostic tables: `edit_history`, `diagnostic_history`, `tool_timing`, `focus_sets`, `focus_state` (1690–1752). Migration strategy: `ALTER TABLE nodes ADD COLUMN...` with `ON CONFLICT` / `INSERT OR IGNORE` for idempotency (1449–1528).

**status-candidate** — **prod-presumed** (running continuously; foundational).

**files** — server.py (entire DB initialization section 1311–1832).

**key constants** — WAL mode enabled (PRAGMA journal_mode=WAL), foreign_keys=ON. User PRAGMA version gates one-shot FTS cleanup migration (1658–1684).

**tests** — test_bootstrap_fresh_db.py, test_engine_flows.py, all integration suites.

## **3. FTS5 Full-Text Search & Indexing**

**what** — Contentless external-content FTS5 virtual table mirrors the core claim/quoted_text/interpretation text fields, indexed for substring and phrase queries. Automatic triggers sync nodes→FTS on INSERT; DELETE; and on UPDATE when status transitions to "retracted" (so retracted nodes drop from search results but superseded nodes remain visible). IDF-based keyword extraction (via `engram_idf.extract_keywords()`) filters high-value query terms to reduce noise.

**how** — FTS table creation in `engram_core.py` DB init. Triggers defined in DB init (nodes_fts_insert, nodes_fts_delete, nodes_retract_remove_from_fts):
- `nodes_fts_insert`: on every INSERT, adds row to FTS.
- `nodes_fts_delete`: on DELETE, removes via magic 'delete' insert.
- `nodes_retract_remove_from_fts`: on status='retracted', removes from FTS (but does NOT remove superseded nodes — Lei's design call 2026-05-22).

Query path: `_search_nodes()` in `engram_query.py:1150` calls `_sanitize_fts_query()` in `engram_core.py:1687` which extracts high-IDF keywords via `engram_idf` module, or falls back to legacy simple tokenization (`_sanitize_fts_query_legacy()` in `engram_core.py:1735`).

**status-candidate** — **prod-presumed** (actively used by surface, query, inspect for text-based retrieval).

**files** — `engram_core.py` (FTS setup in DB init; `_sanitize_fts_query()` :1687; `_sanitize_fts_query_legacy()` :1735). `engram_query.py` (`_search_nodes()` :1150). `engram_idf.py` (keyword extraction).

**key constants** — FTS5 vocab table ensured via `ensure_vocab_table()` call (`engram_core.py:1359`). Retract-trigger PRAGMA user_version gates one-shot migration (DB init section of `engram_core.py`).

## **4. Embedding Manager & Semantic Search**

**what** — Optional lazy-loaded sentence-transformers integration for semantic similarity search via embeddings. EmbeddingManager class auto-downloads the model on first use; embeddings are computed on node creation and stored in the `nodes.embedding` column (JSON array of floats). The sqlite-vec KNN index (vec_nodes virtual table) provides SIMD-accelerated cosine similarity; fallback is O(N) pure-Python cosine distance when sqlite-vec is unavailable.

**how** — `EmbeddingManager` class in `engram_core.py:67`: `__init__()` lazy-loads via sentence-transformers; `embed()` / `embed_batch()` compute embeddings; `cosine_similarity()` is static. Global singleton `_embedder` (`engram_core.py:141`).

`_semantic_search()` in `engram_core.py:2910` computes embeddings from query, then either:
- Uses vec_nodes for KNN: `SEARCH(vec_nodes, embedding, k=...)`
- Falls back to Python loop with `EmbeddingManager.cosine_similarity()` (static method in `engram_core.py:67`)

Embedding stored at creation via `_compute_and_store_embedding()` (called by `_stamp_new_node()` in `engram_core.py:1959`).

**status-candidate** — **prod-presumed** (active when sentence-transformers installed; skipped in CI via ENGRAM_NO_EMBEDDINGS=1 env var to save model download time).

**files** — `engram_core.py` (`EmbeddingManager` class :67; `_semantic_search()` :2910; vec_nodes setup in DB init; `_stamp_new_node()` :1959).

**key constants** — `DEFAULT_EMBEDDING_MODEL = "all-MiniLM-L6-v2"` (`engram_core.py:144`). `_SQLITE_VEC_IMPORT_OK` (`engram_core.py:47–50`), `_VEC_BACKEND_AVAILABLE` (`engram_core.py:64`).

## **5. Memory Tiers, Importance Decay & Recall Surfaces**

**what** — Three-tier memory architecture based on importance scores (confidence × surprise × exponential decay). Tier 1 (working memory, ~200 nodes) surfaces in reflect/checkpoint; Tier 2 (searchable, ~1000 nodes) filters retrieval candidates; Tier 3 (total archive). Exponential decay driven by turn counter: `importance_score = importance_base × decay_base^turn` (default decay_base=1.014). Nodes acquire `importance_base` at creation (from confidence + surprise), then decay exponentially as turns accumulate. Recall events (surface/query/inspect) update `recall_turn` and `recall_count`, boosting the node's importance_score back up.

**how** — Core functions:
- `_compute_importance()` in `engram_core.py:1900`: `base × decay_base^turn`
- `_importance_base_for_node()` in `engram_core.py:1907`: `confidence × (1 + surprise)`
- `_get_tier_threshold()` in `engram_core.py:1919`: dynamic percentile-based threshold, targets tier1_max_nodes=200, tier2_max_nodes=1000
- `_refresh_recall()` in `engram_query.py:576`: re-stamps recalled nodes with current turn's importance
- `_stamp_new_node()` in `engram_core.py:1959`: sets importance_base and importance_score on creation

Columns: `nodes.importance_base`, `nodes.importance_score`, `nodes.recall_turn`, `nodes.recall_count`, `nodes.memory_status` (TEXT: active/faded). Config keys: `memory_config.decay_base`, `current_turn`, `tier1_max_nodes`, `tier2_max_nodes` (from CONFIG_PATH).

**status-candidate** — **prod-presumed** (active in every retrieval; turn counter drives time passage).

**files** — `engram_core.py` (core mechanics: `_compute_importance()` :1900, `_importance_base_for_node()` :1907, `_get_tier_threshold()` :1919, `_stamp_new_node()` :1959; `_get_memory_config()` :1747). `engram_query.py` (`_refresh_recall()` :576; threshold in `_search_nodes()` :1150).

**key constants** — Default decay_base=1.014 (lei 2026-05-19 calibration). Tier 1 max nodes 200, Tier 2 max nodes 1000 (defaults in `engram_core.py:1753`). Decay formula preserves semantic memory indefinitely but down-weights old observations (asymptotic forgetting curve).

**tests** — Implicit in all recall-related tests; explicit decay testing in integration suites.

## **6. Utility Scoring (MemRL-inspired USE model)**

**what** — Reinforcement-learning-inspired utility bump: each node's `utility_score` increases when the node is surfaced in a tool result and later cited as a premise. Uses Q-learning formula `Q_new = Q_old + α(1 - Q_old)` where α is determined by the action's "intensity of impression" (deeper cognitive load = higher α). Three tiers: 0.15 (derive/supersede/contradict), 0.10 (focus), 0.05 (loose touch). Utility_score is composited with semantic similarity in the retrieval ranking formula (multiplicative blend in `engram_query.py:1506–1522`).

**how** — `_utility_reward()` in `engram_core.py:1980` bumps utility for a list of nodes given an action key. `USE_ALPHA` dict in `engram_core.py:215` maps action→α. Reward applied in observation/derivation/supersede/retract/contradiction/focus operations (callers in `engram_derive.py`, `engram_revision.py`, `engram_focus.py`, `engram_observation.py`, etc.). Composited in retrieval via multiplicative amplifiers in `engram_query.py:1506–1522`: `composite = relevance × (1 + UTIL_BETA × utility_score) × (1 + IMP_BETA × normalized_importance)`.

**status-candidate** — **prod-presumed** (active in retrieve/search but effects are subtle; observability poor without utility_score instrumentation).

**files** — `engram_core.py` (`USE_ALPHA` :215; `_utility_reward()` :1980; `UTIL_BETA` :239; `IMP_BETA` :242; `MMR_LAMBDA` :251). `engram_query.py` (composite formula :1506–1522; reward call sites distributed across tool modules).

**key constants** — `USE_ALPHA`: {"derive": 0.15, "supersede": 0.15, "contradict": 0.15, "focus": 0.10, "focus_load": 0.05, "unfocus": 0.05}. `USE_ALPHA_DEFAULT=0.10` (`engram_core.py:236`). `UTIL_BETA=0.10` (`engram_core.py:239`). `IMP_BETA=0.005` (`engram_core.py:242`). (The old UTILITY_AMPLIFIER=0.50 constant was replaced by the UTIL_BETA/IMP_BETA dual-amplifier model in the #872 refactor.)

**tests** — test_utility_use_action.py.

## **7. Confidence Computation & Propagation**

**what** — Single-source-of-truth confidence model (in engram_confidence.py): observations inherit from quote_type (hard_data→0.95, official_statement→0.85, etc.) adjusted by source_class discount (introspective→×0.95, user_stated→override to 0.85). Derivations compute from reasoning_type (deductive→0.98 discount, inductive→0.95, abductive→0.80 with caps) and minimum premise confidence. Axioms are definitionally 1.0; definitions/goals/questions/lessons are non-claim-bearing (no confidence). Full confidence history stored in JSON array on nodes.

**how** — CONFIDENCE_MAP (engram_confidence.py:17–24) hardcodes base confidences by quote_type. SOURCE_CLASS_CONFIDENCE_DISCOUNT (32–36) applies adjustments. REASONING_TYPES (47–66), REASONING_CLASS (69–84), REASONING_DISCOUNT (87–102), ABDUCTIVE_CONFIDENCE_CAP (106–109).

Observation confidence computed in `_add_observation_impl()` in `engram_observation.py:725`; derivation confidence in `_create_derivation()` in `engram_core.py:3604`, `_compute_confidence()` in `engram_core.py:3225`. Validation in `_validate_reasoning_structure()` in `engram_core.py:3550`.

**status-candidate** — **prod-presumed** (every claim-bearing node requires confidence; used in importance calculation, resolution thresholds, supersede decisions).

**files** — `engram_confidence.py` (complete; single source of truth for constants). `engram_core.py` (`_compute_confidence()` :3225; `_create_derivation()` :3604; `_validate_reasoning_structure()` :3550; imports from `engram_confidence`). `engram_observation.py` (`_add_observation_impl()` :725).

**key constants** — CONFIDENCE_MAP values: hard_data=0.95, official_statement=0.85, ..., editorial=0.35. PREDICTIVE_CONFIDENCE_CAP=0.60. CONJECTURE_CONFIDENCE_DEFAULT=0.40 (all in `engram_confidence.py:17–41`). DEFAULT_RESOLUTION_THRESHOLD in `engram_core.py:516`.

## **8. Supersede & Retract Cascade Machinery**

**what** — Immutability via supersession: old versions are never deleted, only marked superseded (is_current→0, superseded_by field set). A new derivation can supersede an older one; retraction removes a node from all query results and the FTS index but preserves it for audit. Cascade deletion is NOT used; instead, edges to superseded/retracted nodes are preserved (ancestors remain visible for forensics). The "cascade" edge relation (`retracts`, `supersedes`) is not confidence-propagating (DAG-exempt).

**how** — Supersede entry: `engram_supersede()` (server.py wrapper at :4552 → `_supersede_impl()` in `engram_revision.py:440`) creates a supersedes edge, updates is_current/superseded_by on old node. Retract entry: `engram_retract()` (server.py wrapper at :4647 → `_retract_impl()` in `engram_revision.py:783`) sets status='retracted', removes from FTS via trigger. Cascade downstream walk: `_walk_cascade_downstream()` in `engram_revision.py:97` traverses edges to find dependent nodes, but does NOT auto-cascade retract/supersede (manual sweep only).

Columns: `nodes.is_current` (1=current, 0=superseded), `nodes.superseded_by` (target node ID), `nodes.status` (active/retracted). Retract trigger fires on status transition into 'retracted' (1625–1635).

**status-candidate** — **prod-presumed** (used in every lifecycle story; retract/supersede are critical cleanup tools).

**files** — `engram_revision.py` (`_walk_cascade_downstream()` :97; `_retract_impl()` :783; `_supersede_impl()` :440). server.py (wrappers: `engram_supersede` :4552; `engram_retract` :4647). `engram_core.py` (retract trigger in DB init).

**key constants** — `DAG_EXEMPT_RELATIONS` in `engram_core.py:506`: {"retracts", "supersedes", "about"} (non-confidence-propagating; derived from EDGE_CLASSIFICATIONS where dag_check=False).

**tests** — test_contradiction_cascade.py, test_engine_flows.py. Supersede/retract coverage in tool payload tests.

## **9. Contradiction Detection & Resolution**

**what** — Explicit contradiction nodes (ct_NNNN) link two conflicting claim-bearing nodes; agent flags the conflict with description + optional root-cause analysis. Resolution (engram_resolve) associates a resolving node (supersession or new derivation) to close the contradiction, marking it resolved. Unresolved contradictions remain visible in reflect/audit. Contradictions themselves are NOT confidence-bearing (confidence=0.5, surprise=1.0).

**how** — `engram_contradict()` (server.py wrapper at :4295 → `_contradict_impl()` in `engram_derive.py:217`) creates ct_NNNN node, inserts contradicts edges to both conflicting nodes, stamps with confidence=0.5 + surprise=1.0. Resolution: `engram_resolve()` (server.py wrapper at :4458 → `_resolve_impl()` in `engram_revision.py:229`) creates resolves edge, marks contradiction status='resolved' or 'partially_resolved' (based on threshold `DEFAULT_RESOLUTION_THRESHOLD` in `engram_core.py:516`).

Query pattern `_pattern_contradiction_obsolescence_ready()` (18206–18290) identifies contradictions past their resolution window for review.

**status-candidate** — **prod-presumed** (active in conflict surfacing; resolution is deliberate, not automatic).

**files** — `engram_derive.py` (`_contradict_impl()` :217). `engram_revision.py` (`_resolve_impl()` :229). server.py (wrappers: `engram_contradict` :4295; `engram_resolve` :4458). `engram_query.py` (`_pattern_contradiction_obsolescence_ready()` :126).

**key constants** — `DEFAULT_RESOLUTION_THRESHOLD` in `engram_core.py:516`.

## **10. Edge Relations & Semantic Bidirectionality**

**what** — Directed edges (source→target, relation) form a DAG with strict relationship types. The edge-classification table (`EDGE_CLASSIFICATIONS` in `engram_core.py:418`) is the single source of truth: each relation type is a dict entry specifying dag_check (bool), removable (bool), and addable_after_creation (bool). Semantically symmetric relations (about, serves, exemplifies) have symmetric queryability even though the underlying edge is unidirectional.

**how** — `EDGE_CLASSIFICATIONS` is a dict keyed by relation name (`engram_core.py:418`) defining every valid relation. Derived constants in `engram_core.py`:
- `DAG_EXEMPT_RELATIONS` (:506): relations treated as non-confidence-propagating (dag_check=False).
- `_ADDABLE_AFTER_CREATION_RELATIONS` (:511): edges addable post-creation via `engram_add_edge`.
- `_REMOVABLE_EDGE_RELATIONS` (:2785): edges removable via `engram_remove_edge`.

Validation: `engram_add_edge()` (server.py wrapper :3432 → `_add_edge_impl()` in `engram_cornerstone.py:474`), `engram_remove_edge()` (server.py wrapper :3346 → `_remove_edge_impl()` in `engram_cornerstone.py:389`) check against `EDGE_CLASSIFICATIONS`. Traversal: neighbors are fetched via `_get_neighbors()` in `engram_query.py:598`, which handles symmetric relation wrapping.

**status-candidate** — **prod-presumed** (every DAG operation validates against this table; extensible design for new relation types).

**files** — `engram_core.py` (`EDGE_CLASSIFICATIONS` :418; derived constants `DAG_EXEMPT_RELATIONS` :506, `_ADDABLE_AFTER_CREATION_RELATIONS` :511, `_REMOVABLE_EDGE_RELATIONS` :2785). `engram_cornerstone.py` (`_add_edge_impl()` :474; `_remove_edge_impl()` :389). `engram_query.py` (`_get_neighbors()` :598). server.py (wrappers: `engram_add_edge` :3432; `engram_remove_edge` :3346).

## **11. Focus Mode (Persistent Pinned Nodes)**

## **12. Nap & Sleep / Turn Advancement**

**what** — Two checkpoint mechanisms: `engram_nap()` persists context WITHOUT advancing the turn counter (used before compaction, end of work burst); `engram_advance_turn()` increments the global turn counter (used only at sleep/session end, triggers exponential memory decay). Both mechanisms snapshot the graph to git (graph_snapshot.md + SQL dump), append to session_log.md, and compute diagnostic metrics. Nap arms a "nap_checkpoint" feeling-report nudge (TTL 5 turns).

**how** — `_checkpoint_internal()` (7620–7771) shared logic: collects stats, appends to LOG_PATH, calls `_commit_snapshot()`, stores diagnostic snapshot in diagnostic_history. Parameter `advance_turn` decides whether to increment the turn counter (`_set_current_turn(new_turn+1)`) or leave it alone.

Entry points:
- `engram_nap()` (7813–7856): calls _nap_impl with advance_turn=False
- `engram_advance_turn()` (7903–7986): calls _advance_turn_impl with advance_turn=True

Nap behavior: mode="nap", turn unchanged, feeling-nudge written via `_write_feeling_nudge()` in `engram_core.py:1775`. Sleep behavior: mode="session", turn incremented, no nudge.

**status-candidate** — **prod-presumed** (daily-once sleep invariant; nap fires multiple times per day at compaction boundaries).

**files** — `engram_lifecycle.py` (`_checkpoint_internal()` :79; `_nap_impl()` :218; `_advance_turn_impl()` :223). server.py (wrappers: `engram_nap` :1951; `engram_advance_turn` :2065).

## **13. Git Snapshot Layer (_commit_snapshot)**

**what** — Durable version-controlled record of graph state: at each checkpoint (nap/sleep), the graph is serialized as a human-readable Markdown snapshot (graph_snapshot.md) and a full SQL text dump (knowledge.sql). Both are committed to the local git repo (~/.engram/.git) with the checkpoint message as the commit log. This provides a restore point for schema migrations, audit trail for graph evolution, and rollback mechanism for errors.

**how** — `_generate_snapshot()` (4333–4549) walks all nodes grouped by type, renders each as Markdown sections with status/confidence/edges. `_commit_snapshot()` (4552–4776) calls _generate_snapshot, writes SNAPSHOT_PATH, computes SQL dump, checks out git, stages specific files (graph_snapshot.md, knowledge.sql, session_log.md, config.json, warm-briefing.md), and commits with the provided message. Best-effort: never raises; git errors are logged but don't block the checkpoint.

Git setup: `_init_git()` (748–790) initializes ~/.engram/.git if absent. `_git()` helper (721–743) runs git commands via subprocess. Conditional: `_git_available` flag (746) is False if git init fails or ENGRAM_DISABLE_GIT env var is set.

**status-candidate** — **prod-presumed** (active at every checkpoint; critical safety mechanism).

**files** — `engram_core.py` (`_generate_snapshot()` :2357; `_commit_snapshot()` :2576; `_init_git()` :578; `_git()` helper :551).

**key constants** — `GIT_EXE` (`engram_core.py:542`); `GIT_TIMEOUT=15s` (`engram_core.py:548`; temporarily bumped to 30s during init); git config user email/name set in `_init_git()` (`engram_core.py:578`).

## **14. WAL Guard (#786) — Displacement Detection & Degraded Marker**

**what** — SQLite WAL (Write-Ahead Log) self-guard detects shm-file displacement (deletion or atomic replacement while the server holds an open fd). Incident: 2026-06-03, knowledge.db-shm was deleted under a live MCP server, causing split-brain (stale WAL index) for 6 hours. Detection is Linux-only (/proc/self/fd/ inode comparison); writes a degraded-state marker to .substrate-degraded.json when displacement is detected, enabling downstream processes to coordinate recovery.

## **20. Recall Surfaces (surface/inspect/query/subgraph/list)**

**what** — Five asymmetric query surfaces with different purposes:
- `surface()`: shallow nudge (type counts, special nodes, top claims, age signal; no refresh)
- `inspect()`: deep single-node with three view modes (recall=neighbors, deep=fields+history, edges=topology)
- `query()`: keyword+semantic search with result ranking (relevance + importance amplifier + utility)
- `subgraph()`: neighborhood walk (N hops from a source, returns adjacency)
- `list()`: type-filtered scan (all nodes matching type filter, sorted by importance)

All but `surface()` update the node's recall_turn + recall_count (refresh memory). Dream-mode flag skips refresh side-effect for maintenance reads.

**how** — Entry points (all follow _<tool>_impl() pattern):
- `engram_surface()` (6668–7026): aggregates counts + topological hints.
- `engram_inspect()` (7028–7213): single-node three-view formatter.
- `engram_query()` (7215–7402): multi-result ranker via _search_nodes() + tiered result builder.
- `engram_get_subgraph()` (7405–7619): graph walk with distance limit.
- `engram_list()` (17819–18129): type-filtered scan with recall refresh.

Core search: `_search_nodes()` (6118–6570) implements FTS + semantic search, tiering, ranking. Result builder: `_build_tiered_results()` (6571–6667) formats results with recall_summary/keywords fallbacks.

**status-candidate** — **prod-presumed** (core discovery mechanism; heavily exercised).

**files** — server.py (6668–7619 surface/inspect/query/subgraph; 17819–18129 list; 6118–6667 search/rank core).
